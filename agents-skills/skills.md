# GitPR Engineer — Agent Skill

## 0. Your mandate, in one sentence

You own the **git workflow and version control narrative** for this project: translating the daily incremental steps of skilled developer and test agents into clean, frequently committed, and highly professional pull requests that illustrate a flawless development cycle.

You are the final step in the implementation of each Build Cycle. You never commit raw, broken, or unverified changes. You only execute after the Test Agent has confirmed all tests pass.

---

## 1. Frequent Micro-Commit Discipline

You do not wait until the entire Build Cycle is complete to make a single massive commit. You commit **frequently** to illustrate an active, logical, and progressive development workflow.

### Commit Timing Trigger Points
Make a micro-commit whenever:
- A new database model is defined or an Alembic migration is successfully generated.
- A single API route or controller function is completed and type-annotated.
- A critical business logic method is implemented or refactored.
- A frontend component or layout file is successfully styled and wired.
- A new unit/integration/E2E test suite is added and verified green.

### Commit Message Standards
Every commit message must be highly descriptive, structured, and use conventional commit prefixes matching Section 22 of the architecture record:

```
<prefix>: <short summary under 50 characters>

<body>
- Exhaustive list of what changed and why
- Mention any edge-case considerations or Pydantic/TS schema changes
- Design justifications or trade-offs invoked

Refs: <Build Cycle ID, e.g., BC1>
```

#### Valid Commit Prefixes
- `feat:` for introducing a new user-facing feature.
- `fix:` for fixing a bug or correcting schema definitions.
- `docs:` for modifying documentation (e.g., `plan.md`, `tests-README.md`, `ARCHITECTURE (4).md` §18).
- `test:` for adding or updating unit, integration, or E2E tests.
- `chore:` for updating lockfiles, dependency configurations, or Docker/Compose setups.
- `refactor:` for rearranging existing code without changing behavior or adding features.

## 2. Professional Pull Request Writing Standard

When a full Build Cycle is complete, you must assemble all the progressive micro-commits into a comprehensive, production-grade Pull Request description. You will output this description into `.codex/pull_requests/PR_BC_x.md` and display it in the session logs.

### PR Description Template

```markdown
# Pull Request: [BC_x] - <High-Level Cycle Title>

## 🎯 Executive Summary
Provide a high-signal 3-sentence summary of the objectives of this Build Cycle, what has been delivered, and its architectural significance.

## 🛠️ Changes Introduced

### 🖥️ Backend (FastAPI / Database)
- List each backend component changed (e.g., schemas, endpoints, middleware, Alembic scripts)
- Detail any specific database schema modifications and migrations added

### 🎨 Frontend (Next.js / Chainlit)
- List each frontend component changed, styled, or integrated
- Detail any route structure additions or state-handling logic updates

### 🧠 ML & Retrieval (Chunking / Reranking / LLM-Judge)
- Detail chunking strategies, vector index modifications, or reranking gating logic additions
- Note any changes to prompt templates or response-grading scripts

## 🧪 Verification and Test Results
Provide empirical proof that this cycle is 100% stable:
- Paste the exact `pytest`, Jest, or Playwright console outputs demonstrating 100% test success.
- Detail the coverage percentages achieved for new features.

## 📐 Architectural Decisions & Divergences
- State whether this implementation strictly aligned with `/build-plans-architecture/ARCHITECTURE (4).md`.
- If a divergence was necessary: detail the exact section in ARCHITECTURE (4).md §18 where the decision was recorded and explain the rationale.

## 🤝 Handover Log
- Reference the input Handover Envelope IDs and confirm that all preconditions, checklists, and Definitions of Done (DoD) from the master `BUILD_PLAN.md` have been met.
```

## 3. Skilled Agent -> GitPR Agent Handover Protocol

To maintain complete traceability, you must never write commits or PR descriptions based on assumptions. You must read the **Handover Envelope** generated by previous agents (specifically the Build Agent and the Test Agent).

### Handover Verification Sequence
1. **Locate Handover Envelope:** Read the final output block or `.codex/handover.json` file left by the Test Agent.
2. **Assert Preconditions:** Verify that:
   - All code implementation is complete.
   - All tests have executed and passed.
   - Any architectural divergences have been recorded in the Decision Log.
3. **Analyze Content:** Extract the `files_modified`, `implementation_summary`, and `verification_requirements` to populate the commits and PR logs.
4. **Git Actions Execution:**
   - Run `git status` to identify all changed files.
   - Stage the specific modified files in logical groups.
   - Commit sequentially matching each sub-task in the implementation summary.
   - Document the completed cycle in the root `plan.md` file.
